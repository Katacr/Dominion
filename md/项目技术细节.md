## 1. 前期研究

### 1.1 TUI/CUI 架构

研究文件：`TUI-CUI-Architecture-Research.md`

已确认：

- 当前 22 个 `AbstractUI` 页面分别实现 TUI/CUI，业务回调大量重复。
- TUI 通过随机动态 SecondaryCommand 保存闭包 callback。
- CUI 通过玩家 UUID、slot 和 ChestButton 闭包保存 callback。
- 两者最终大多调用相同的 `*Command` 或 Provider。
- 新架构不能继续复制第三套 Dialog 业务分支。

### 1.2 Spigot Dialog

研究文件：`/home/plugins/API/SpigotDialogAPI/Spigot-Dialog-API-Research.md`

已确认：

- Spigot 与 Paper 使用不同 Dialog API 和 callback 数据模型。
- 共享层不能暴露任一平台的 Dialog 类型。
- Dialog renderer 必须按平台延迟加载。

### 1.3 列表菜单

研究文件：`TUI-CUI-List-Inventory.md`

已确认：

- TUI 有 17 个真正动态列表，CUI 有 18 个真正动态列表。
- Flag 设置列表共有环境、访客、成员、权限组、模板 5 类。
- 动态列表还包括领地、成员、玩家候选、权限组、模板、称号、复制来源和 Residence 迁移。
- 固定工具栏不能继续作为第一条列表 entry 占用分页容量。

## 2. UI 重构蓝图

蓝图文件：`UI-Refactor-Blueprint.md`

核心结构：

```text
uis/_shared/<menu-id>.yml
uis/<locale>/text_menu/<menu-id>.yml
uis/<locale>/chest_menu/<menu-id>.yml
uis/<locale>/dialog_menu/<menu-id>.yml
```

约束：

- `_shared` 保存 route、provider 和统一 action groups。
- 语言/类型菜单只保存布局和表现。
- callback 只传递短期 token，可信业务参数保存在服务端 session。
- ActionExecutor 严格顺序执行，失败和导航结果终止旧序列。

## 3. 第一版实现范围

状态：第一版已完成，待服务器手动测试。

第一版只实现一个不影响现有菜单的纵向切片：

1. 平台无关的 ActionSpec、ActionParser、ActionRegistry、ActionExecutor 和 ActionResult。
2. `tell/actionbar/title/toast/hovertext` 反馈动作。
3. Dominion 白名单 action registry，并提供少量旧 UI 过渡 action。
4. TUI `Layout + Buttons` YAML 加载、校验和渲染。
5. 玩家绑定、过期、一次性消费的固定 callback session。
6. `/dominion ui_menu action_demo` 样板入口。
7. 不迁移 MainMenu，不实现动态 LIST，不改变现有 TUI/CUI 分流。

选择独立样板入口的原因：Action 和 TUI 底座必须先经过 Spigot/Paper 实测，不能在第一版直接替换玩家现有主菜单。

- 内联动作使用按钮 id 作为 callback 审计标识，共享动作保留 action-group id；两者都使用相同玩家绑定 token。
- 不包含共享动作组的菜单允许省略 `_shared.action-groups`，便于纯展示型菜单只使用内联反馈动作。

## 8. TUI 与 Actions 基底加固

状态：已完成，待服务器手动测试。

本阶段只加固 TUI 与跨 renderer 共用的 Actions/session/router，不引入 CUI 或 Dialog 平台实现。后续 renderer 必须直接复用本节契约。

### 8.1 原子 reload 与 revision

- `TuiMenuRepository.load()` 在临时 map 中完成全部安装、解析和校验，成功后才原子替换当前菜单快照并递增 revision。
- `Configuration.loadConfigurationAndDatabase()` 成功完成常规配置加载后同步 reload `uis/**`。
- UI YAML reload 失败只报告错误，旧菜单快照和旧 revision 继续可用，不影响 Dominion 其他配置。
- reload 成功后 `UiCallbackSessionManager.activateRevision()` 清空全部旧 callback。
- callback 同时保存 menu revision；旧 token 和 reload 竞态中迟到注册的 callback 都不能执行。
- `UiCallbackSessionManager` 使用两阶段启动：菜单首次校验成功后才注册 Bukkit listener 和清理定时任务，初始化失败不会残留 hook。

### 8.2 Session 安全

- token 继续使用 128-bit 随机 UUID 文本、玩家 UUID 绑定、一次消费和默认 5 分钟有效期。
- 到达过期时刻即视为失效，不额外延长一个清理周期。
- 跨玩家提交不会移除原玩家 token。
- revision 切换与 callback 注册使用同步临界区，避免 clear/register 交错留下可执行旧 token。
- 自动测试覆盖一次消费、跨玩家、精确过期时刻、revision 失效及旧 revision 注册拒绝。

### 8.3 Action 加载期校验

- `ActionRegistry` 现在注册 `ActionHandler + ActionValidator`，未知类型和错误参数在 YAML load 时拒绝。
- `title/toast/open/page/refresh/close/hovertext` 均校验必填参数、允许字段和控制流约束。
- 单个动作最长 4096 字符，一个 actions 列表最多 32 项。
- `hovertext` 必须闭合 tag、包含 text，并在 actions/url/copy 中恰好选择一个 click target；URL 只允许 HTTP(S)。
- `DominionActionRegistry` 同样注册 operation validator，未知 `dominion:` operation 不再等到玩家点击才失败。
- Template operation 当前不接受 YAML 参数，模板目标只从 callback trusted arguments 获取。

### 8.4 执行线程与超时

- `ActionExecutor` 保持严格串行，只有 `CONTINUE` 进入下一动作。
- 每个 handler 都通过 `EntityActionDispatcher` 在触发玩家所属线程开始执行。
- Spigot 主线程或 Folia 当前已拥有 region 时直接执行，避免每个短 action 强制多等待一 tick。
- 异步 handler 完成后，下一 action 会再次经过 entity dispatcher，不继承任意 completion thread。
- 整个 action workflow 默认最多 30 秒；超时返回带 `TimeoutException` 的 FAILURE，防止未来 Dialog 等待状态永久挂起。

### 8.5 异步渲染顺序

- `UiRequestTracker` 为每个玩家分配单调 request id。
- 同一玩家连续打开页面时，旧 provider 即使最后完成，也不能覆盖较新的 route。
- 旧请求完成不会错误移除新请求；请求结束后 tracker 条目按 request id 条件删除。
- 新 route 开始加载时立即使旧可见菜单 callback 失效，避免加载窗口重复执行旧操作。
- provider 完成后再次检查玩家在线状态、request id 和 menu revision，再进入 entity scheduler 渲染。

### 8.6 动态文本安全

- Layout、按钮、hover、tell、actionbar、title、toast 和 hovertext 均使用 MiniMessage unparsed placeholder 插入动态变量。
- 配置文件中的颜色和格式继续生效，entry/player/route 变量中的 `&`、MiniMessage tag 和 click tag 只作为字面文本。
- clickable URL 在动态变量解析后再次执行 HTTP(S) 校验。
- Toast 使用预构建 Component 转换为 legacy section text，不再对动态变量执行第二次 `&` 颜色解析。

### 8.7 当前自动测试边界

新增覆盖：

- 内置 YAML 的完整语义加载，而不只是 YAML 语法读取。
- Action 参数 validator、Dominion operation validator、数量限制和 workflow timeout。
- 玩家请求顺序以及旧请求完成不影响新请求。
- clickable text 的动态文本和动态 URL 安全。
- session 玩家绑定、一次消费、过期和 revision 生命周期。

## 9. SelectTemplate 与 EnvFlags 迁移

状态：已完成，待服务器手动测试。

### 9.1 共享 route schema

- `_shared` 菜单支持 `route.required` 和 `route.optional`。
- route key 必须满足统一命名规则，不能同时声明为 required 和 optional。
- `UiRouter` 在 provider 执行前拒绝缺失 required 参数或携带未声明参数的 route。
- `ActionContext.requireRouteArgument` 和 `requireTrustedArgument` 为所有 domain operation 提供统一失败语义。

### 9.2 SelectTemplate

新增 provider 与 operation：

```text
templates.member-apply-candidates
dominion: apply-member-template
dominion: back-member-flags
```

route：

```text
menu-id: select_template
required: dominion.name, member.name
```

行为：

- provider 打开页面前重新验证领地管理员权限和目标成员存在。
- 模板名保存在 entry trusted arguments，不进入点击命令或普通 route 文本。
- 含 `ADMIN` 的模板只向领地主人开放 apply operation，普通管理员看到 disabled 按钮。
- apply operation 再次查询模板、领地和成员，并根据模板 ADMIN 状态重新执行 owner/admin 鉴权。
- 应用成功后返回配置化 `MemberFlags` route，不再产生新旧 TUI 往返。
- 旧 `SelectTemplate.showTUI` 只负责构建可信 MenuRoute；CUI 和配置化故障回退不变。

## 10. 通用输入与 TUI/CUI 聊天适配

状态：已完成首个纵向切片，待服务器手动测试。

### 10.1 分层边界

输入不强制三种 renderer 使用同一种交互组件：

```text
TUI/CUI chat Inputter  -> ChatInputWorkflow -> InputSchema
Dialog form response   -> Dialog adapter    -> InputSchema
                                           -> ActionContext.submittedInputs
                                           -> shared domain operation
```

- `InputSchema`、`InputFieldSpec` 和 `ActionContext.inputs` 为 renderer-neutral。
- `ChatInputWorkflow` 只属于需要聊天输入的 TUI/CUI，不会成为 Dialog 依赖。
- Dialog 后续直接将输入组件响应交给同一个 InputSchema，再构造携带 inputs 的 ActionContext。
- 业务 operation 只能通过 `requireInput` 读取提交值，不能把 `{input.*}` 当作可信资源 id。

### 10.2 InputSchema

支持类型：

```text
TEXT
INTEGER
DECIMAL
BOOLEAN
OPTION
```

规则：

- schema 最多 16 个字段，field key 使用 route/input 统一命名规则。
- 单字段最长 1024 字符，并支持 required、trim、长度、pattern 和 option whitelist。
- INTEGER、DECIMAL、BOOLEAN、OPTION 转换为稳定 canonical string。
- 拒绝 renderer 提交的未声明字段。
- `InputValidationException` 只记录 field key 和安全错误，不在日志或消息中回显完整提交值。

### 10.3 ActionContext inputs

- `submittedInputs` 与 route arguments、entry trusted arguments 分开保存，信任等级不混合。
- `withInputs` 保留 player、route、revision、trusted arguments、action groups 和 callback registrar。
- normalized input 以 `{input.<key>}` 合并到展示 variables，动态文本仍通过 unparsed placeholder 渲染。
- domain operation 使用 `requireInput(key)` 进行显式读取和二次业务校验。

### 10.4 ChatInputWorkflow

- 当前只接受 single-field schema，适配现有“一次聊天输入一个值”的 Inputter 模型。
- 输入校验失败发送安全错误并重新注册 prompt，不丢失原 route/trusted context。
- 输入 `C` 取消后返回 `REFRESH`，重新打开原配置化 route。
- 默认等待输入 2 分钟，提交业务 future 最多等待 30 秒。
- timeout 只取消仍为当前 runner 的 workflow，旧 timeout 不会移除后来替换的新 runner。
- submission completion 通过玩家 entity scheduler 交给统一 ActionResult handler。

### 10.5 旧 Inputter 并发修复

- pending runner 从 `HashMap<Player, ...>` 改为 `ConcurrentHashMap<UUID, ...>`。
- AsyncPlayerChatEvent 使用原子 remove 实现一次消费，两个并发聊天事件只有一个能取得 runner。
- runner 在异步事件线程取出后，通过 `Scheduler.runEntityTask` 在玩家所属线程执行。
- `unregister` 使用 key/value 条件删除，旧 runner 无法删除替换后的新 runner。
- 玩家退出按 UUID 清理 pending runner。

### 10.6 创建模板输入样板

- 配置化 `TemplateList` 的 `create-template-input` 不再调用旧 `CreateTemplateInputter`。
- schema 定义 `template.name` 为 1 至 64 字符、trim、不可包含空白的 TEXT。
- 提交后 operation 通过 `context.requireInput("template.name")` 读取值，再检查权限和重名。
- 创建成功返回 `REFRESH`；校验失败重试，取消或超时返回原模板列表。
- 旧 CUI 以及 rename/message/map-color/resize 等过渡型操作继续使用原 Inputter，不受此样板影响。

### 10.7 TUI 与 provider 资源上限

- 每个 Layout 最多 64 行，单行最多 4096 字符。
- 每个 Buttons scope 最多 128 个按钮，每行最多 32 个按钮 component。
- 菜单最多 256 个变量，按钮文本、hover、URL、COPY 和变量值最多 4096 字符。
- button/menu/operation id 最长 64 字符。
- MenuEntry display/trusted map 各最多 64 项，单值最多 4096 字符，entry id 最长 256 字符。
- provider 返回条目数仍不得超过 renderer 请求的 LIST rows。

```text
CuiRouter
  -> CuiRenderer
  -> CuiInventoryHolder
  -> CuiListener
  -> UiCallbackDispatcher
  -> UiResultRouter(CUI)
```

- `CuiRenderer` 第一版只接受无 dynamic source 的菜单；遇到动态菜单会明确失败，动态 provider/page 能力留到迁移 `DominionList` 时实现。
- renderer 根据 9 字符 Layout 构建 Inventory；重复 symbol 会复用表现定义，但每个 `slot + click` 注册独立 opaque token。
- ItemStack 支持 material/name/lore/amount/custom-data/glow；变量替换后再进入 PlaceholderAPI 和 legacy color 处理。
- Paper 使用 Adventure ItemMeta，Spigot 使用 legacy `setDisplayName/setLore`，沿用项目现有跨核心兼容方式。
- glow 通过稳定的 `minecraft:luck_of_the_sea` registry key 查找，不直接链接可能随 Bukkit 版本改名的静态字段。
- `item-model` 通过反射 capability adapter 调用新 Bukkit API；旧核心仅告警一次并忽略该表现字段，不产生类加载错误。
- permission 不满足时不会注册 callback；`hidden-when-disabled: true` 时槽位为空，否则保留只读 ItemStack。

生命周期与导航：

- `ConfiguredTuiManager` 注册 `CuiListener` 和 CUI result handler，共用现有 session、executor、actions、provider registry。
- `OPEN` 优先打开目标 configured CUI；目标 chest 文件尚未迁移时，临时关闭 Inventory 并打开同 route 的 configured TUI，保证业务按钮可用。

- `MainMenu` 已验证静态布局、权限隐藏、共享 action group、UI 切换和管理员操作链路；下一页按实际需求迁移 `DominionList` 并补动态 source/pagination。

## 15. 第一版 Spigot Dialog 框架

### 15.1 源码复核结论

- Spigot 公开入口为 `Player.showDialog(net.md_5.bungee.api.dialog.Dialog)` 与 `Player.clearDialog()`，不需要直接发送 NMS 数据包。
- 服务端按钮使用 `CustomClickAction`，提交通过 `PlayerCustomClickEvent` 返回；输入字段和按钮 additions 合并为 Gson object。
- Spigot 当前公开 Body 只有 `PlainMessageBody`，但服务端 Dialog codec 和客户端支持 `minecraft:item`；适配器通过受控的自定义 `DialogBody` JSON 模型补齐该能力。
- KaMenu 的 `Title/Settings/Body/Inputs/Bottom` 层级可复用，但 Paper builder、`DialogResponseView`、Events、条件、JavaScript 和 Tasks 不进入 Dominion 第一版。

### 15.2 模块与类加载边界

```text
core/uis/menu/dialog
  -> platform-neutral definitions, loader and InputSchema

dialog-spigot
  -> Bungee Dialog model
  -> PlayerCustomClickEvent
  -> Gson payload adapter
```

- `core` 继续以旧 Bukkit API 编译，公开接口不出现 Bungee Dialog、Gson 或高版本 Bukkit event 类型。
- `dialog-spigot` 使用 Java 21 和 Spigot 1.21.6 API 编译；插件只在非 Paper 且事件类存在时反射实例化 adapter。
- Paper 当前不加载 Spigot adapter，后续必须提供独立 Paper renderer，不能假定 Paper 含 Bungee Dialog API。

### 15.3 第一版配置能力

- 顶层复用 KaMenu 命名：`Title`、`Settings`、`Variables`、`Body`、`Inputs`、`Bottom`。
- `Settings` 支持 `can_escape`、`pause`、`after_action`；默认使用 `NONE` 保持当前 Dialog，兼容读取旧的 `CLOSE`，暂不支持 `WAIT_FOR_RESPONSE`。
- `Body` 支持 `type: message`，以及 KaMenu 基础 `type: item` 字段：`material/amount/description/width/height/show_overlays/show_tooltip`。
- `Inputs` 支持 KaMenu 的 `input`、`slider`、`dropdown`、`checkbox`，提交后统一进入 `InputSchema` 再写入 `{input.<key>}`。
- `Bottom` 支持 `notice`、`confirmation`、`multi`；按钮使用 `action-group` 或内联 `actions`，严格二选一。
- 不支持 Events、条件、JavaScript、Tasks、repeat button 和 Item Body；动态 Provider 列表已在业务菜单迁移阶段补齐。

### 15.4 Callback 与测试入口

- Spigot 按钮 additions 只携带 `_dominion_token`，菜单 ID、动作路径、权限和 route 不由客户端提交。
- adapter 按玩家保存当前 Dialog 的 token 集合；首次有效提交会立即失效其余兄弟按钮，阻止关闭后的 callback 重放。
- `UiCallbackSessionManager` 可为 token 保存可选 `InputSchema`；`UiCallbackDispatcher` 在消费后拒绝额外字段并规范化字符串值。
- callback 继续执行同一 `ActionExecutor`、action groups 与 `UiResultRouter(DIALOG)`，不复制 Dominion 业务 handler。
- 双语言样板资源位于 `uis/<locale>/dialog_menu/action_demo.yml`。
- 管理员测试入口为 `/dominion ui_dialog action_demo`；第一版不会改变玩家 TUI/CUI 偏好或 `/dominion` 默认入口。

## 16. 第一版 Paper Dialog 适配器

### 16.1 版本边界

- Mojang 在 1.21.6 引入 Dialog 协议，但 Paper `1.21.6-R0.1` API 尚未公开 Dialog builder。
- KaMenu 已在 Paper 1.21.7 实测可用，因此 `dialog-paper` 以 Paper `1.21.7-R0.1` 和 Java 21 作为最低编译基线。
- 启动时同时探测 `io.papermc.paper.dialog.Dialog` 与 `DialogResponseView`；缺失时只禁用 Dialog，不影响 TUI/CUI。
- Paper 1.21.7 尚无 `Audience.closeDialog()`；普通回调使用 `after_action: NONE` 并直接替换表单，支持 `closeDialog()` 的新版本才可执行显式关闭。

### 16.2 Paper 映射

- `DialogBase`、`DialogBody.plainMessage`、四类 `DialogInput` 与三类 `DialogType` 直接映射同一中立定义。
- 文本使用 `LegacyToMiniMessage.parseTemplate`，变量以 unparsed placeholder 插入，避免配置变量注入 MiniMessage 标签。
- `DialogResponseView` 按 input definition 读取 text、float 或 boolean，再转换为字符串交给共享 `InputSchema`。
- 每个 Paper `DialogAction.customClick` 仍捕获 opaque token；callback options 固定一次使用、五分钟有效。
- 首次按钮提交立即失效同一 Dialog 的兄弟 token，与 Spigot adapter 和 CUI holder 保持相同的重放防护。

### 16.3 平台选择

```text
Paper + Dialog API -> dialog-paper
Spigot + Dialog API -> dialog-spigot
其他/旧版本       -> Dialog disabled, TUI/CUI unchanged
```

- 两个 adapter 均实现 core 的 `DialogPlatformAdapter`，最终 JAR 不打包 Paper、Spigot、Bungee 或 Gson API 类。
- `/dominion ui_dialog action_demo` 在 Paper 与 Spigot 上读取相同双语言 YAML、action groups 和输入 schema。

### 16.4 Spigot 反馈动作加固

- Spigot Action Bar 显式使用 `Player.Spigot.sendMessage(ChatMessageType.ACTION_BAR, BaseComponent...)`，避免 Adventure Bukkit facet 在新 Spigot 核心上选择不兼容路径。
- Toast title 使用 `GsonComponentSerializer` 生成完整组件 JSON。
- 纯 Spigot 26.2 使用 `ClientboundUpdateAdvancementsPacket` 添加客户端临时 advancement，并在 10 tick 后发送 remove packet；不会写 datapack、修改服务端 advancement registry 或执行资源重载。
- Spigot 26.2 数据包 transport 初始化失败时明确拒绝发送，不回退到会调用 `PlayerList.reloadResources()` 的 `UnsafeValues.loadAdvancement`；Paper 和旧版本暂时保留原兼容路径。
- Dialog action FAILURE 现在会记录底层 cause，后续 Spigot API 差异不再只显示玩家消息而没有后台堆栈。

## 17. Dialog 业务菜单迁移

### 17.1 Provider 列表解析

- 新增平台中立 `DialogDynamicDefinition`，配置使用 `Dynamic.source/page-size/empty/layout/width/buttons` 描述 Body 列表行和行内动作模板。
- `DialogMenuResolver` 直接复用 TUI/CUI 的 `MenuDataRegistry`、`MenuDataProvider`、`MenuEntry` 与 `PageSlice`，不在 Paper/Spigot adapter 内复制查询逻辑。
- Provider 输出的 `entry.*` 只用于显示；`trustedArguments` 只绑定到服务端一次性 token，不写入客户端提交数据。
- 每个 entry 只展开其 `availableActionGroups` 允许的按钮，Residence 子节点等只读条目不会获得越权操作回调。
- `DominionCopySourceProvider` 与 `UngroupedMemberProvider` 的可用集合使用共享 action-group ID `copy`/`add-member`，而不是底层 operation 名；该约定同时适用于 TUI、CUI 和 Dialog。
- 动态页最多 32 个 entry、每个 entry 最多 4 个行内动作；动态条目进入 `Body.plainMessage`，footer 只保留分页、返回和固定命令。
- `previous/next` 在首页或末页由 resolver 移除；Provider 修正越界页码后，callback 使用修正后的 route。

### 17.2 刷新生命周期

- 内置 Dialog 使用 `after_action: NONE`；普通按钮和行内文本不会先关闭表单，动作完成后延迟 1 tick 重新解析 Provider 并直接替换当前 Dialog，避免闪烁和鼠标回到屏幕中心。
- 导航动作返回 `OPEN`，原地修改返回 `REFRESH`；两者均延迟 1 tick 执行，使数据库和缓存回调的结果在重绘前可见。`CLOSE` 仅用于显式关闭、进入旧聊天输入或切换 UI。
- CUI 的 `OPEN`、`REFRESH`、`STOP` 和失败恢复同样延迟 1 tick；刷新前再次校验玩家仍打开同一路由的 Dominion inventory，避免过期回调覆盖玩家后来打开的界面。
- Title 与 Residence 迁移命令新增“不重开旧菜单”的兼容重载；旧命令入口保持原行为，配置化 UI 改由统一结果重绘。
- 已迁移页面中的返回动作不再直接调用旧 `showTUI/showCUI`，统一返回 `MenuRoute`，避免 Dialog 回调打开错误 renderer。

### 17.2.1 DUI 偏好与界面一致性

- `PlayerDTO.UI_TYPE` 新增 `DUI`（Dialog User Interface）；玩家选择存入 `PlayerDOO.ui_preference`，并由 `PlayerRepository.updateUiPreference` 写入玩家表的 `ui_preference` 字段。
- 三种主菜单只提供一个“界面类型”入口，统一打开 `ui_preferences` 二级菜单；该页面再提供 TUI/CUI/DUI 选择，避免主菜单堆叠多个切换按钮。
- `Configuration.defaultUiType` 仍决定全局策略：`BY_PLAYER` 时读取玩家持久偏好，固定为 `TUI/CUI/DUI` 时覆盖个人选择；基岩版 UUID 仍优先使用 CUI。
- `AbstractUI` 的 DUI 分支复用各旧入口已经完成的 `MenuRoute` 参数构建，`ConfiguredTuiManager.show` 再将该 route 交给 Dialog renderer；因此无需在 22 个旧 UI 类内复制 route 映射。
- 持久化偏好只决定从命令或旧入口首次打开哪种界面；按钮 callback 仍以 token 中的 `UiSurface` 为准，Dialog 回调不会因为玩家数据库里曾保存 CUI 而跳转到箱子菜单。
- 配置化动作不得调用会自动执行 `displayByPreference` 的旧命令。缓存/配置重载和复制命令新增“不重开旧菜单”入口；Dialog 删除领地直接等待 Provider Future 完成，再由原 surface 刷新。

### 17.2.2 原生二级输入

- `main_menu.create`、`dominion_manage.rename`、`dominion_manage.enter-message`、`dominion_manage.leave-message` 和 `dominion_manage.map-color` 已配置 `input-capture`，Paper 与 Spigot 均显示原生二级输入 Dialog。
- Mojang 26.2 的 Dialog input name 不接受点号和连字符；表单提交键统一使用 `snake_case`，路由参数仍保留 `dominion.name` 等点号命名。
- `DialogMenuLoader` 在资源加载时校验原生 input name，避免非法键直到玩家点击二级表单时才由 Paper/Spigot 抛错。
- action 根据 `submittedInputs` 区分来源：Dialog 输入通过共享 `InputSchema` 规范化后直接调用 Dominion Provider；TUI/CUI 没有提交字段时继续启动原聊天输入器。
- 创建、重命名、进入提示语、离开提示语和地图颜色均等待 Provider Future 完成后返回 `OPEN/REFRESH`，再延迟 1 tick 重绘，确保数据库和缓存结果已经可见。

#### 动态 locale 发现

- `uis/manifest.yml.locales` 仅定义随 JAR 安装的内置语言；仓库加载时还会扫描 `plugins/Dominion/uis/<locale>/` 下所有包含菜单子目录的有效 locale。

- 自定义语言允许只翻译部分菜单；缺失文件回退到 `en_US`，无效或已删除的玩家 locale 回退到服务器全局语言，再回退到 `en_US`。

#### 渲染边界

- `UiRouter`、`CuiRouter` 和 `ConfiguredTuiManager.showDialog` 每次打开 route 时通过玩家缓存解析 locale，因此语言修改在下一次刷新立即生效。

## 18. Dialog 二级表单与 Body 可点击列表

### 18.1 Body 列表约束

- Provider 列表不得把每个 entry 展开为原生 Bottom 按钮，避免大量宽按钮破坏 Dialog 布局。
- `Dynamic.layout` 使用 `{entry.*}` 展示状态、名称和说明，并用 `{按钮模板ID}` 插入可点击文本，例如 `&e[{entry.state}] &f{entry.name}: &7{entry.description} {toggle}`。
- `DialogMenuResolver` 将当前页全部 entry 用换行拼接为一个带 `inlineButtons` 的 `DialogBodyDefinition`；不会为每个条目创建独立 message Body。
- 每行按钮占位符会重写为 `entry-<index>-<button-id>`，避免多行复用 `{toggle}` 时 callback 互相覆盖；不可用 action-group 的占位符直接移除。
- Bottom 只保留 previous、next、back、exit 与少量固定操作。
- Paper 行内动作使用 Adventure `ClickEvent.callback`；Spigot 复用 TUI 已验证的 `RUN_COMMAND` token 传输，避免 Bungee serializer 强制要求 `ClickEventCustom` 子类。
- Spigot 普通行内动作执行隐藏的 `/dominion ui_callback`；带 confirmation/input-capture 的行内动作先执行 `/dominion ui_dialog_callback`，由 adapter 打开二级表单后再消费原 token。
- 行内 `tooltip` 在 Paper 映射为 Adventure hover，在 Spigot 映射为 Bungee `HoverEvent.SHOW_TEXT`。

### 18.2 通用二级表单

- 任意 Dialog 按钮可配置 `confirmation` 或 `input-capture`，两者严格互斥。
- 两类二级表单均复用 `title/body/confirm/deny`；确认与取消按钮支持 `text/tooltip/width`。
- `input-capture.input` 复用顶层 Inputs 的 `input/slider/dropdown/checkbox` 解析和 `InputSchema` 校验。
- 打开二级表单时原业务 token 不执行；确认后才消费，取消后销毁 token 并重新打开父 Dialog。
- confirmation 的可信标记由 adapter 直接传给 `UiCallbackDispatcher`，不会接受客户端伪造的普通输入字段。
- `tooltip` 支持单行字符串或 KaMenu 风格字符串列表，列表按换行组合。

### 18.3 首批业务接入

- confirmation：删除领地、删除权限组、删除模板和批量 Residence 迁移。
- input-capture：创建/重命名权限组、创建/重命名模板、创建领地以及修改进入/离开提示语；TUI/CUI 继续使用现有聊天输入 workflow。
- 领地重命名和地图颜色已经接入同一 `input-capture`；尺寸输入仍保留原输入器，待其业务写入逻辑从旧 Inputter 解耦。
- 双语言动态菜单已统一改为 Body 行内 `[管理]`、`[切换]`、`[删除]` 等操作文本。

### 18.4 验证

## 19. Spigot 26.2 Item Body 与 Toast

### 19.1 Item Body 适配

- `DialogBodyDefinition` 以平台中立字段保存 message/item 两类 Body；Loader 对 material、1-99 数量和 1-256 尺寸进行边界校验。
- Paper adapter 使用官方 `DialogBody.item(...)`；Spigot adapter 使用 `SpigotItemDialogBody` 继承 Bungee `DialogBody`，让现有 Gson serializer 输出 Mojang codec 所需的 `minecraft:item` JSON。
- Spigot 方案不链接 NMS。已用服务器实际 `VersionedComponentSerializer` 验证生成字段为 `item.id/count`、可选 `description`、`show_decorations`、`show_tooltip`、`width` 和 `height`。
- 当前只覆盖 KaMenu 的基础 material item，不包含 name、lore、components、player head 或 slot reference；这些属于后续增强，不应在 Loader 中静默接受。

### 19.2 Toast 数据包适配

- `ToastTransport` 隔离 core 与版本化实现；只有非 Paper 的 Spigot 26.2 反射加载 `SpigotToastTransport`。
- transport 使用 `Advancement.CODEC + JsonOps` 解析临时 impossible advancement，构造已完成的 `AdvancementProgress`，再通过玩家连接发送 `ClientboundUpdateAdvancementsPacket`。
- advancement ID 使用随机 `dominion:ui_toast_*`，只存在于客户端；10 tick 后发送仅含 removed ID 的 packet，不产生服务器文件和 registry 残留。
- 反射签名已对本机 Spigot 26.2 服务端 jar 执行 `javap` 核对。版本升级后必须重新检查 Identifier、packet 构造器、progress 和 connection send 契约。

### 19.3 本地验证

- `DialogMenuLoaderTest` 通过，覆盖 Item Body 基础字段。
- `:core:compileJava`、`:dialog-paper:compileJava`、`:dialog-spigot:compileJava` 和完整 `shadowJar` 均通过。
- 最终 lite JAR 已确认包含 `ToastTransport`、`SpigotToastTransport`、`SpigotItemDialogBody` 和 `DialogBodyDefinition`；本轮未部署或重启测试服，实际客户端渲染与 Toast 展示仍需后续游戏内回归。
